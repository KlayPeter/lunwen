## 2. 项目真实依据

后续所有图都应围绕下列真实实现来画，避免画成泛化架构图。

### 2.1 前端真实模块

- 主体框架：`ccb/src/components/chat/ChatView.vue`
- 私聊/群聊内容页：`ccb/src/views/Content.vue`
- 聊天室页：`ccb/src/views/ChatRoom.vue`
- 收藏页：`ccb/src/views/Favorites.vue`
- 路由：`ccb/src/router/index.js`
- 状态：`ccb/src/stores/useChatStore.js`
- 侧边栏与列表：`Sidebar.vue`、`LastChats.vue`、`Contacts.vue`、`ChatRoomListPanel.vue`
- AI 面板与总结：`AIAssistantPanel.vue`、`SummaryDialog.vue`、`AIExplainDialog.vue`
- 聊天室组件：`CreateChatRoomDialog.vue`、`JoinChatRoomDialog.vue`、`ShareRoomDialog.vue`、`ChatRoomDetail.vue`
- 消息相关组件：`ChatMessageList.vue`、`ChatInput.vue`、`MessageActions.vue`、`QuotedMessage.vue`、`CodeMessage.vue`
- 移动端入口：`BottomNavbar.vue`

### 2.2 桌面端真实模块

- Electron 主进程：`ccb/main.js`
- Electron 预加载：`ccb/preload.js`
- 桌面特性：托盘、未读角标、消息预览、系统通知、窗口管理

### 2.3 后端真实模块

- 服务入口：`server/server.js`

- 收藏：`server/controllers/FavoriteController.js`、`server/models/Favorite.js`
- AI 助手：`server/controllers/AIController.js`
- 聊天室 AI：`server/controllers/ChatRoomAIController.js`
- Agent 总结与问答：`server/controllers/AgentController.js`
- 上传与文件：`server/controllers/UploadController.js`、`server/uploads`

### 2.4 数据与检索真实模块

- 用户：`server/models/Users.js`
- 私聊消息：`server/models/Messages.js`
- 群聊/聊天室消息：`server/models/GroupMessage.js`

## 3. 第2章图片详细说明

### 图2-1 系统技术架构图

- 图类型：分层架构图。
- 论文作用：说明项目从接入端到前后端再到数据与 AI 层的真实技术关系，是第 2 章的总图。
- 必须体现的项目事实：
- Web 端和 Electron 桌面端共享一套前端业务逻辑。
- 前端以 Vue 3、Pinia、Vue Router、Axios、Socket.IO Client 为核心。
- 后端以 Express + Socket.IO 提供 API 和实时通信。
- 数据核心是 MongoDB，不是复杂微服务架构。
- AI 能力来自 DeepSeek 接口、MessageIndexer、RAGService、VectorSearchTool。
- 推荐布局：
- 最好采用横向或纵向四层结构。
- 第一层：接入层。
- 第二层：前端实现层。
- 第三层：后端服务层。
- 第四层：数据与智能层。
- 必须出现的标签：
- 接入层：`Web端`、`Electron桌面端`、`系统通知/托盘`
- 前端层：`Vue 3`、`Vue Router`、`Pinia`、`Axios`、`Socket.IO Client`、`ChatView`、`Content`、`ChatRoom`、`Favorites`
- 后端层：`Express API`、`JWT认证`、`Passport OAuth`、`Socket.IO`、`ChatController`、`UserController`、`FavoriteController`、`AIController`、`ChatRoomAIController`、`AgentController`
- 数据与智能层：`MongoDB`、`Users`、`Messages`、`GroupMessage`、`Favorite`、`AIConversation`、`Summary`、`Uploads`、`MessageIndexer`、`RAGService`、`VectorSearchTool`、`DeepSeek API`
- 画面说明：
- 这张图不能画成互联网大厂的云原生架构，也不能加 Kafka、Kubernetes、分布式网关一类与项目主实现不匹配的元素。
- 图上要让人一眼看出三条主线：普通业务请求链路、实时通信链路、AI 检索增强链路。
- 模块颗粒度保持中等，不要细到每个 API 路径，也不要粗到只剩“前端”“后端”“数据库”三个盒子。

### 图3-2 系统业务流程图

- 图类型：泳道流程图。
- 论文作用：从业务角度说明用户、客户端、服务端、存储/检索之间如何形成闭环。
- 推荐泳道：
- `用户`
- `Web/Electron客户端`
- `业务服务层`
- `AI/RAG/存储`
- 必须出现的真实链路：
- 登录进入系统。
- 前端加载会话列表、历史消息、在线状态。
- 服务端做身份校验并返回数据。
- 用户发送文本、代码、图片或文件。
- Socket.IO 推送事件。
- MongoDB 持久化。
- MessageIndexer 加入索引队列。
- 若人工未解决，则调用 AI / RAG。
- AI 回答回写消息流或总结记录。
- 收藏与标签备注形成知识沉淀。
- 画面重点：
- 这张图要体现 HTTP 和 Socket.IO 是并行存在的。
- 也要体现 AI 与检索不是单独漂在外面，而是接在业务闭环里面。
- 可直接用于出图的描述：

```text
绘制一张中文泳道式系统业务流程图，四个泳道依次为用户、Web/Electron客户端、业务服务层、AI/RAG/存储。流程从用户登录并进入交流空间开始，客户端加载会话列表、历史消息和在线状态，服务层校验身份并返回上下文数据。随后用户发送文本、代码、图片或文件，业务服务完成校验、Socket.IO 广播与 MongoDB 持久化，MessageIndexer 将文本消息加入索引队列。若人工回复不足，则触发 AI/RAG 检索与问答，AI结果回写消息流或总结记录。最后用户对关键内容执行收藏、标签备注和再次检索。整体强调 HTTP、Socket.IO、MongoDB、RAG 与知识沉淀之间的闭环关系。
```

- 主流程方向优先从左到右或从上到下，不要来回折返。
- 盒子、箭头、泳道、系统边界、UML 元素保持同一套线宽和圆角风格。
- 字号以论文打印可读为准，宁可删减次要标签，也不要缩小到难辨认。
- 同一章的图尽量保持同一视觉语言，避免章内风格飘散。

## 8. 使用建议

- 最好一张一张生成，不要一次性让模型同时生成 17 张图。
- 如果生成结果过于空泛，就把对应图号的“必须出现的标签”和“对应项目事实”一起喂给模型。
- 如果生成结果信息过密，就优先删减次要字段和辅助说明，不要继续缩字号。
- 结构图优先保证“关系正确”，流程图优先保证“顺序清楚”，用例图和时序图优先保证“角色边界明确”。
- 第 4 章图片相对更偏设计视角，结构应比第 3 章更工整、更像论文图，而不是更花。
