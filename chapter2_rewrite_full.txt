\chapter{相关技术}

\section{从需求出发的技术组合}

先解决什么？不是把功能一股脑铺满，而是抓住技术交流里最常见的动作：发消息、同步状态、管理房间、回看历史。问题一旦落到这些场景，技术选择就会自然收紧。系统因此同时覆盖基础聊天能力和扩展能力，前者包括消息发送、状态同步与历史存储，后者涉及桌面端封装、AI 问答和阶段性总结。前端采用 Vue 3、Pinia、Vue Router 组织页面与状态；后端用 Express 暴露接口，再由 Socket.IO 处理实时事件；MongoDB 承担核心存储，AI 部分通过大语言模型接口配合检索增强接入。这个组合谈不上追新，更像是在当前体量下做了一次克制的取舍。

图2-1展示了系统的整体分层：接入层、前端应用层、服务层，以及数据和智能辅助层。Web 端与 Electron 端共用一套前端业务逻辑，避免同一套会话流程维护两份代码；常规业务经由 HTTP 访问，私聊、聊天室消息和在线状态则交给 Socket.IO 长连接处理；服务端收到请求后，一边执行业务逻辑，一边把消息写入 MongoDB，并在需要时调用大语言模型与检索组件生成回答或讨论摘要。结构并不花哨，但路径清楚，现阶段的主要需求基本都能被覆盖。实际开发里，网络偶尔抖动时更能看出这种分层的价值，常规接口和实时链路分开后，问题定位会轻一些。

这套架构不是凭空抽象出来的，代码目录能一一对上。前端主体放在 \texttt{ccb/src}，桌面端主进程与预加载脚本分别位于 \texttt{ccb/main.js} 和 \texttt{ccb/preload.js}；服务端入口是 \texttt{server/server.js}，用户、聊天、聊天室、收藏和 AI 相关能力各自拆成独立路由与控制器；模型集中在 \texttt{server/models}，消息索引和检索逻辑则落在 \texttt{server/services} 与 \texttt{server/tools}。换句话说，图2-1总结的是当前实现，不是为了画图再另起一套说法。

\section{前端实现技术}

聊天界面为什么不适合随便拼页面？因为它同时挂着消息列表、联系人区、聊天室区、输入面板、收藏详情和 AI 侧栏，模块之间牵连很多，但又不能写成一个“大组件”。基于这一点，系统把 Vue 3 作为前端核心框架，重点看中的就是组件化组织和响应式更新能力。组合式 API 也更顺手，状态逻辑、事件处理和界面渲染可以围着业务模块展开，不必把所有代码挤进单个页面文件里\cite{vue3reactive2023,frontend2024}。实际调试时，如果连续收到十几条消息，先观察的往往不是理论性能指标，而是滚动区域是否抖动、输入框是否被误刷新。

围绕 Vue 3，项目再补上 Pinia、Vue Router 和 Vite。Pinia 负责登录状态、当前会话对象和跨组件共享信息；Vue Router 管理聊天页、聊天室页与收藏页的切换；Vite 处理开发和构建流程。这样搭起来后，消息一到，界面可以跟着响应式状态直接重绘，不需要手工维护一层又一层 DOM；另一边，Web 页面迁到 Electron 时也更省事，重复开发的部分明显少了。现有代码里，\texttt{ChatView.vue}、\texttt{Content.vue}、\texttt{ChatRoomDetail.vue} 与 \texttt{Favorites.vue} 构成了主要交互入口，这一点和论文描述是一致的。

桌面端怎么办？项目没有再做一套原生客户端，而是引入 Electron 复用既有前端代码。这样处理后，托盘驻留、消息角标、最近消息预览、通知跳转这些桌面能力放到主进程补齐即可，渲染层继续专注界面本身。对当前项目来说，这种方案更节省时间，也压住了实现复杂度，同一套业务逻辑服务多端的思路也更容易落地。

\section{后端服务与实时通信技术}

服务端部分选择 Node.js 和 Express，原因其实很直接：聊天系统里既有数量不少的短请求，也有持续存在的长连接，Node.js 的异步事件模型比较适合处理这种混合负载；Express 的路由结构又足够清晰，用户、消息、聊天室、收藏和 AI 功能可以按模块拆开实现\cite{nodejs2023,architecture2023,restful2024}。项目没有再往里叠更重的编排层和中间层，主要逻辑仍放在控制器、模型与服务模块。这样做不算炫技，但调试路径短，对本科阶段项目也更实际。

实时通信是另一条关键链路，这里依赖 Socket.IO。只靠 HTTP 轮询行不行？能做，但不合适。新消息到达、成员进出聊天室、输入状态变化、在线人数更新，这些场景都要求服务端主动把事件推给客户端。Socket.IO 在原生 WebSocket 之上补齐了自动重连、事件命名和房间广播，私聊与群聊两类场景因此更容易组织\cite{websocket2023,socketio2024}。当前实现中，\texttt{server/sockets/chat.js} 处理私聊事件，\texttt{server/sockets/room.js} 负责聊天室与群消息广播，边界分得比较明确。遇到连接反复重试的情况时，这种拆分也更利于定位是哪一类事件出了问题。

认证部分采用 JWT。用户登录成功后，服务端把用户标识写入令牌返回客户端，后续请求再携带该令牌完成权限校验。这个方案把会话管理压得更轻，也更贴合前后端分离系统的常见实现方式\cite{jwt2023,oauth2024}。从现有代码看，邮箱密码登录、邮箱验证码登录和 Google OAuth 登录最终都会汇入同一套用户模型与认证流程，后面的好友关系、聊天室成员身份和收藏权限也因此有了统一入口。

\section{数据存储与消息组织技术}

数据层为什么选 MongoDB？原因不复杂：聊天消息本身就不整齐。普通文本、图片、文件、代码片段、系统通知、AI 回复都可能落在同一类消息记录里，收藏、AI 对话和总结记录又各有字段差异。面对这种结构，文档型数据库比严格的关系表更灵活，后续扩字段时也不会太别扭，所以更贴合本项目的实现需要\cite{mongodb2023,nosql2024}。

在具体建模上，项目把用户、聊天室、私聊消息、群聊消息、收藏、AI 对话和聊天总结拆成独立集合，没有为了“抽象得漂亮”硬做统一大表。为什么这么分？访问路径本来就不一样。用户数据主要支撑登录、好友和身份展示；私聊消息与聊天室消息对应两套查询方式；收藏与总结更偏向知识沉淀。再配合索引和分页，系统在当前规模下可以较快完成消息读取、历史检索和收藏回看。这里还有一个比较实用的小处理：上传文件统一放在 \texttt{server/uploads}，数据库里只通过 \texttt{fileInfo} 字段关联文件信息，避免大文件直接写库；前端拿到消息类型后，也能按图片、文档或其他附件分别渲染。实际维护时，这种目录式存放比把二进制内容塞进文档里省心得多。

\section{AI辅助技术}

本系统里的 AI 不是聊天之外的装饰功能，它直接嵌在技术讨论流程里。普通 AI 对话由 \texttt{AIController} 处理，聊天室里的问答、代码分析、相似问题检索和总结生成则交给 \texttt{ChatRoomAIController}。两类控制器都通过 HTTP 接口调用外部大语言模型完成推理，因此本地不必部署完整模型，重点转到提示词管理、上下文拼装和结果回传上。这种做法把部署门槛压低了不少，也更符合毕业设计通常能拿到的算力和环境条件\cite{llm2024,transformer2023}。

如果 AI 只盯着用户刚发出的那一句话，会怎样？最常见的问题就是答得快，却和聊天室上下文脱节。为了解决这个问题，系统加入了消息索引与检索增强机制。`MessageIndexer` 会把可用文本消息放进索引队列，代码里默认按 10 条一批、每 5000 ms 处理一次，尽量不阻塞正常发消息；`VectorSearchTool` 在聊天室问答前先检索历史讨论，再把检索结果和当前问题一起整理成提示内容发给模型。外部 embedding 服务可用时，系统走向量检索；未配置时则退到本地 hash embedding；如果向量库暂时未就绪或没有结果，还会继续用关键词搜索兜底\cite{rag2024,vectordb2023}。精度当然会有差别，不过本地环境至少能跑通，这一点在开发阶段很重要。

从实际用途看，AI 在系统里主要做三件事。第一，围绕技术问题给出即时说明，尽量缩短用户等人工回复的时间。第二，把聊天室阶段性讨论压缩成摘要，后来加入的成员不用从头翻完整记录。第三，把历史消息纳入检索范围，让生成结果尽量贴近房间里已经形成的讨论脉络。和泛用聊天机器人相比，这种组合更贴近技术交流群的真实使用场景，也更符合 Mini-Chat-Bar 的设计目标。
