系统测试是在系统实现完成之后开展的综合性验证工作，其目的并不只是判断各个功能能否运行，还需要进一步确认前后端交互、实时通信、AI 接口调用以及桌面端封装后的行为是否符合设计预期。Mini-Chat-Bar 同时包含 Web 端、Electron 桌面端、Express 接口层、Socket.IO 实时通信层和 MongoDB 数据层，测试范围因此不能停留在单一模块，而必须围绕完整业务链路展开。基于这一考虑，本章从测试环境、功能测试和性能测试三个方面对系统进行说明，并结合项目中的真实接口、脚本和运行结果，对测试过程、测试结果和结果分析进行系统整理。

\section{测试环境与测试方法}

\subsection{测试方法与策略}

本次测试采用黑盒测试与脚本辅助验证相结合的方式开展。黑盒测试主要面向系统外在行为，重点检查登录、聊天、聊天室、收藏、AI 问答和桌面端通知等功能是否能够按照用户预期完成；脚本辅助验证则用于缩短重复操作时间，并对接口耗时、Socket 广播时延和资源占用进行量化记录。测试工具并未脱离项目本身单独设计，而是直接围绕现有接口和脚本展开，例如 AI 相关链路参考了项目中的 \texttt{server/test-chatroom-ai.js} 与 \texttt{server/test-ai-quick.js}，实时通信测试则使用 \texttt{socket.io-client} 建立多个本地客户端连接，对 \texttt{login}、\texttt{private-message}、\texttt{join-group} 和 \texttt{group-message} 等事件进行验证\cite{testplatform2024,blackboxtest2024,webobservability2024}。

测试策略遵循“先模块、后集成；先核心链路、后补充分支”的顺序。具体而言，首先确认后端服务和数据库连接是否正常，以 \texttt{/api/user/test} 路由作为烟雾测试入口；随后围绕用户认证、房间创建、消息发送和收藏建立最小可运行数据；在此基础上继续验证 Web 端界面行为、Socket 事件同步和 AI 返回链路；最后再进行重复请求和多客户端并发测试。这样的安排有一个直接好处，即每一组性能数据都能追溯到具体业务动作，而不是脱离系统上下文的抽象压测结果。测试流程完成后，对通过项、异常项和可改进项分别进行记录，为章节最后的测试总结提供依据。

\section{功能测试}

\subsection{功能测试用例设计原则}

功能测试用例的设计以项目当前已经实现并在代码中可明确定位的业务链路为依据，重点覆盖用户认证、私聊与聊天室消息、AI 问答与总结、收藏管理以及桌面端消息提醒五类核心功能。之所以将这几部分作为优先对象，是因为它们恰好贯穿了前端页面、后端接口、Socket 实时通信、数据库读写以及桌面端封装的主要交互路径。如果这些链路运行稳定，系统的主体功能就具备了基本可用性；如果其中任一链路出现问题，用户往往会立即感知到异常。

在设计原则上，本次测试强调完整性、独立性、可重复性和可追踪性。完整性要求测试用例既覆盖正常操作，也覆盖错误输入、边界条件和状态切换；独立性要求单个用例可以在不依赖前一用例结果的前提下单独执行；可重复性要求相同环境下重复执行应得到一致结果；可追踪性要求每个测试项都能映射到具体接口、页面组件或事件机制。基于这一原则，测试用例表中统一给出前置条件、操作步骤、预期结果和实际结果，使每一条结论都能够回溯到明确的验证过程。

\subsection{接口与实时通信测试}

\subsection{功能测试结果总结}

综合以上结果，系统的主要功能链路已经达到毕业设计阶段可以交付和演示的要求。用户认证、私聊消息、聊天室消息、AI 问答、AI 总结以及桌面端通知等关键功能都完成了既定任务，其中登录、消息发送和消息读取这些高频操作表现稳定，说明系统核心业务骨架已经建立完成。尤其是在接口与实时通信联合测试中，HTTP 请求和 Socket 事件没有出现相互冲突的情况，这一点对聊天系统尤为重要。

与此同时，测试也暴露出系统尚未完全收口的部分。最典型的问题出现在收藏模块的组合筛选路径，一次带关键字的类型筛选返回了 500 状态码，这说明当前收藏查询逻辑在复杂条件下仍缺少充分的异常校验。AI 模块虽然能够返回有效结果，但在新建聊天室样本数据较少时，RAG 历史来源不会明显体现，说明其表现依赖于更丰富的历史语料积累。总体而言，系统功能层面的可用性已经得到验证，后续优化重点应放在收藏复杂筛选的健壮性、AI 召回命中率以及更大规模回归测试的自动化程度上。

\subsection{性能测试结果总结}

综合响应时间、并发性能和资源占用三部分结果可以得出一个比较明确的结论：在当前单机部署条件下，Mini-Chat-Bar 已经能够满足课程项目、小型团队交流和毕业设计答辩演示的使用需求。普通业务接口全部保持在毫秒级返回范围内，消息发送与消息读取过程顺畅，100 个并发 Socket 客户端的广播测试也未出现丢包和掉线情况，说明系统在核心聊天能力上已经具备稳定性。

当然，性能测试也揭示了当前系统的真实边界。首先，AI 问答与总结的耗时明显高于本地业务链路，这并不是简单的代码优化就能彻底解决的问题，而是外部大模型调用带来的客观等待。其次，本章的并发测试仍然建立在单节点和本地网络条件之上，尚未覆盖公网弱网、跨地域访问和多实例部署等更复杂场景。再次，收藏模块组合筛选出现的异常说明，性能之外的稳定性问题同样值得关注。后续若继续完善系统，可从三个方向着手：一是为 AI 高频问题引入结果缓存和异步队列，缩短用户等待时间；二是扩充历史语料并优化向量检索配置，提升 RAG 的召回效果；三是补充针对复杂查询条件和更高并发规模的回归测试，使系统在功能和性能两方面都更接近可持续运行的软件产品。
