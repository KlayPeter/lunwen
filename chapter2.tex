\chapter{相关技术}

\section{技术选型与架构概述}

Mini-Chat-Bar 的技术路线不是先把功能全部铺开，再去寻找实现方式，而是先从技术交流场景里最常见的需求出发，再据此收敛技术选择。系统既要完成消息发送、状态同步、房间管理和历史存储这些基础能力，也要兼顾桌面端封装、AI 问答和聊天总结等扩展功能。基于这一点，前端采用 Vue 3、Pinia 和 Vue Router 组织页面与状态，后端采用 Express 提供接口服务，并借助 Socket.IO 处理实时事件，数据层以 MongoDB 为核心存储，AI 能力则通过大语言模型接口与检索增强方式接入\cite{vueguide2026,piniadocs2026,expressdocs2026,socketiodocs2026,mongodbdocs2026,gpt4report2023,ragllmsurvey2024}。这样的组合没有刻意追求“新”或“全”，但比较适合当前项目的目标和复杂度。

如图2-1所示，系统整体技术架构由接入层、前端应用层、服务层以及数据与智能辅助层组成。Web 端与 Electron 桌面端共享同一套前端业务逻辑；前端通过 HTTP 接口访问用户、房间和收藏等常规服务，并通过 Socket.IO 长连接完成私聊、聊天室消息和在线状态同步；后端在处理业务请求的同时，把消息写入 MongoDB，并在需要时调用大语言模型和检索组件生成 AI 回答或讨论总结。整套架构并不复杂，但层次关系比较清楚，能够覆盖本项目现阶段的主要需求。

\section{前端实现技术}

前端部分采用 Vue 3 作为核心框架，主要考虑在于它的组件化组织方式和响应式机制与聊天系统的界面特点比较匹配。聊天界面里同时存在消息列表、联系人列表、聊天室列表、输入面板、收藏详情和 AI 侧边区域，这些部分既彼此关联，又不能完全耦在一起。Vue 3 的组合式 API 更适合把不同功能拆成可复用组件，让状态逻辑、事件处理和界面渲染围绕具体业务模块展开，而不是全部挤在单个页面文件中\cite{vueguide2026,piniadocs2026}。

在 Vue 3 的基础上，系统全面引入了现代化的前端工程化工具链。其中，使用 Pinia 替代传统的 Vuex 来管理全局登录态、当前激活的会话对象和部分跨组件共享信息。Pinia 不仅去除了繁琐的 Mutation 结构，还提供了对 TypeScript 推导的内置支持，使数据流向更为清晰扁平；配合 Vue Router，系统能够高效组织聊天页、聊天室详情页和个人收藏页之间的动态路由拦截与鉴权切换。不仅如此，项目摒弃了老旧 Webpack，转而以 Vite 作为底层开发与构建载体\cite{vueguide2026,piniadocs2026,electrondocs2026}。得益于 Vite 基于原生 ESM 的热更新（HMR）机制，项目在调试复杂聊天界面时展现出了极速的冷启动和局部模块重载能力。这样的技术栈组合带来了两个比较直接的好处：消息到达后界面能够依赖 Vue3 的 Proxy 响应式更新机制迅速完成虚拟 DOM 重绘，免去了大量手工维护边界状态的烦恼；同时，标准化的前端架构也保证了 Web 页面能够“原封不动”且较为顺畅地迁移到 Electron 沙箱中进行复用，极大摊薄了多端客户端的重复迭代成本。在最终的实际代码库中，\texttt{ChatView.vue}（骨架层）、\texttt{Content.vue}（消息滚动控制区）、\texttt{ChatRoomDetail.vue} 和 \texttt{Favorites.vue} 紧密配合，共同构成了系统前端敏捷轻量的主交互入口。

Electron 的引入主要解决桌面端使用场景。相比重新开发一套原生客户端，Electron 允许系统直接复用既有前端代码，并在主进程中补充托盘驻留、消息角标、最近消息预览和通知跳转等桌面特有能力\cite{electrondocs2026}。对本项目而言，这一方案比原生桌面开发更符合时间和复杂度约束，也能较好体现“同一业务逻辑、多端复用”的思路。

\section{后端服务与实时通信技术}

后端部分以 Node.js 为运行环境，以 Express 为接口框架。Node.js 的异步事件模型适合处理聊天系统中大量短请求与长连接并存的情况，而 Express 提供了比较清晰的路由组织方式，使用户管理、消息处理、聊天室管理、收藏管理和 AI 功能可以按模块拆分实现\cite{expressdocs2026,socketiodocs2026}。在本项目中，后端并没有引入过重的服务编排或复杂中间层，而是把主要逻辑保持在控制器、模型与服务模块中，这样既便于调试，也符合本科阶段项目的实现和维护方式。

实时通信能力依赖 Socket.IO 完成。和单纯基于 HTTP 的系统相比，聊天系统更看重“服务器主动通知客户端”的能力，例如新消息到达、成员进入或离开聊天室、输入状态变化、在线人数更新等场景都不适合用轮询方式频繁触发。Socket.IO 在原生 WebSocket 基础上补充了自动重连、事件命名和房间广播机制，使私聊和群聊两类场景的实现更直接\cite{socketiodocs2026,imsecurity2024,imwebsockets2024}。在当前项目中，\texttt{server/sockets/chat.js} 负责私聊相关事件，\texttt{server/sockets/room.js} 负责聊天室和群消息广播，这种拆分方式让两类通信逻辑在代码层面保持了相对清晰的边界。

在系统安全性与无状态交互设计上，身份认证部分果断拥抱了 JWT（JSON Web Token）令牌机制。用户一旦成功通过校验，服务端将生成含有核心用户标识、签发时间及有效期的加密令牌返回给客户端。此后，客户端在所有的 HTTP Axios 请求以及 WebSocket 握手阶段，均需在头部携带该令牌以完成边界拦截与解析校验。这种“服务端不保存 Session 状态”的做法，不仅免除了内存维护开销，也天然满足了系统未来潜在的分布式横向扩容（Scale-Out）需求，更契合纯粹前后端分离工程的当代规范\cite{oauthbcp2025,oauthcn2024}。从当前项目骨架的实现来看，无论是基于传统的邮箱密码登录、基于 Nodemailer 驱动的验证码快速登录，还是基于外部 OAuth 授权回调的第三方登录，最终在网关层都被强行“拉平”收口到统一的一套 JWT 颁发清洗流中。这种高内聚的防腐层设计，不仅彻底隔离了下游业务对多维度验证源的感知，更为后续跨域调用接口、确认聊天室成员身份体系以及数据权限控制奠定了坚不可摧的底层基石。

\section{数据存储与消息组织技术}

本系统的数据持久层抛弃了较重的传统关系型范式，选用 MongoDB 作为数据承载的中枢骨架。相较于普通 MIS 平台，即时聊天应用产出的数据在结构上表现出极强的“非一致性”波动。同样的一条消息模型单元，在某个瞬间它可能是普普通通的纯文本，在下一秒它又可能是内嵌高亮版式的代码片段，抑或是挂载了独立元数据（名称、哈希、路径指向）的图片包与离线附件；与此同时，“知识网摘”式的收藏快照、树状分化出的跨域 AI 语境记录以及长文本聚合拉出的讨论总结，它们内部的嵌套深度和字段繁简度更是千差万别。如果强行把这批业务丢进关系表并用外键联结，势必会制造出无数毫无意义的 NULL 留白碎片和拖慢读写性能的表相交查询。相比之下，MongoDB 原生依靠 BSON 特性驱动的“文档式/弱模式（Schema-Free）”生态在这个场景下展现出了降维打击般的结构弹性生存能力\cite{mongodbdocs2026}。它不仅能用近乎零负担的代价从容吸纳随时可能延伸的业务扩展字段，还能利用其灵巧且强大的复合索引（Compound Indexes）结构配合 WiredTiger 引擎的底层写入锁，迎合系统这种高频度、巨量按“聊天室 Room + 时间戳排序”聚合刷屏读取的复杂 I/O 操作脉冲，成为本项目幕后极为强韧的数据基底。

具体到数据组织方式，项目把用户、聊天室、私聊消息、群聊消息、收藏、AI 对话和聊天总结分别建模为独立集合。这样的设计并没有刻意追求高度抽象，而是围绕真实访问路径展开：用户信息主要服务于登录、好友和身份展示；私聊消息与聊天室消息分别对应两类查询模式；收藏和总结则承担知识沉淀功能。结合索引设置和分页查询方式，系统能够在当前规模下较快地完成消息读取、历史检索和收藏回看。除此之外，上传文件统一落在 \texttt{server/uploads} 目录下，通过消息记录中的 \texttt{fileInfo} 字段与数据库内容建立关联，这样既避免把大文件直接写入数据库，也便于前端根据消息类型选择展示方式。

\section{AI辅助技术}

系统中的 AI 能力并不是独立于聊天过程之外的附加项，而是围绕技术讨论中的高频需求组织起来的。当前实现中，普通 AI 对话由 \texttt{AIController} 负责，聊天室内的问答、代码分析、相似问题查找和总结生成则由 \texttt{ChatRoomAIController} 负责。两类控制器都通过 HTTP 接口调用外部大语言模型服务完成推理，因此系统不需要在本地部署完整模型，只需管理提示词、上下文和返回结果即可。这种方案降低了运行门槛，也更符合本科毕业设计的资源条件\cite{gpt4report2023,llmsurvey2023,qaoverview2025,agentoverview2024,smartchat2025}。

在技术交流场景中，如果 AI 只依据单次提问生成回答，很容易忽略聊天室已有讨论内容，回答质量也会受到影响。为此，系统引入了消息索引与检索增强机制。项目中的 \texttt{MessageIndexer} 会把可用文本消息加入索引队列，\texttt{VectorSearchTool} 在聊天室问答时优先检索历史讨论，再把检索结果与当前问题一起组织成提示内容发送给模型。当前实现允许在未配置外部 embedding 服务时退化到本地 hash embedding 模式，这虽然在语义精度上不如完整向量模型，但能够保证系统在本地环境下顺利运行，并为后续扩展留下接口\cite{ragllmsurvey2024,ragsurveycn2025,longcontext2024,graphrag2025cn,deepseekrag2026}。

从应用效果看，AI 技术在本系统中的角色主要体现在三个方面：围绕技术问题提供即时问答与解释，降低用户等待人工回复的时间成本；对聊天室中阶段性的讨论进行摘要整理，帮助后续成员快速进入上下文；把历史消息纳入检索范围，使 AI 的回答尽可能贴近当前聊天室已经形成的讨论内容。与通用聊天机器人相比，这种能力组合更贴近技术交流群体的使用方式，也更符合本系统的设计目标。

如图2-2所示，整个智能问答拆分为离线/异步的“消息索引”与在线实时的“检索增强问答”两个环节。当用户发送内容后，系统在后台对消息进行切片与向量化提取并送入向量库（Hash Embedding）；当触发问答调用时，系统通过 `VectorSearchTool` 迅速召回最相关的前 Top-K 历史讨论，再将其组装成 Prompt 发送给第三方 DeepSeek API 获取精准结案。这套设计有效解决了大模型直接回答容易偏离当前讨论上下文的问题。
